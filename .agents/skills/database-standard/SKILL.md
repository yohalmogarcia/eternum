# Database Development Standards

Your goal is to ensure all database schemas, SQL scripts, and JPA entities strictly adhere to the naming, structure, and integrity rules.

## 1. Naming Conventions

### Table Prefixes
- **Transactions:** Direct descriptive name (e.g., `invoice_header`, `receiver`).
- **Temporary Tables:** Use `temp_` prefix (e.g., `temp_invoice_header`).
- **System Tables:** Unique descriptive name (e.g., `company`, `users`).
- **Company Parameters:** Use `company_system_parameter`.

### Column Naming
- **Primary Keys:** `pk_` followed by the singular table name (e.g., `pk_product`).
- **Foreign Keys:** `fk_` followed by the referenced table name (e.g., `fk_company`).
- **Tax Authority Codes (MH):** Use suffix `_mmhh` (e.g., `code_mmhh`, `description_mmhh`).
- **Audit Fields:** Use `created_date`, `updated_date`, `created_by`, and `updated_by`.
- **Boolean Flags:** Use prefixes `is_`, `has_`, or `can_` (e.g., `is_active`).

### Reserved Words
- **NEVER use PostgreSQL reserved words** for table names, column names, or any database object identifiers (e.g., `user`, `order`, `table`, `primary`, `foreign`, `check`, `references`).
- **User/Account Tables:** Always use plural form `users` instead of `user` (which is a PostgreSQL reserved word).
- **Order/Transaction Tables:** Always use descriptive names like `orders`, `transactions`, `requests` instead of reserved words.
- When in doubt, consult the PostgreSQL reserved words list and append an appropriate suffix or use synonyms.

## 2. Standard Data Types

| Use Case | Correct Type | Constraints / Notes |
|----------|-------------|---------------------|
| Identifiers | `integer` | Use `GENERATED BY DEFAULT AS IDENTITY`. |
| Money/Financial | `numeric(16,2)` | NEVER use `float` or `real`. |
| Short Strings | `varchar(n)` | NO `null` allowed; default to empty string `''`. |
| Long Text | `text` | For descriptions or large content. |
| Date/Time | `timestamp without time zone` | Or `date` as appropriate. |
| Complex Data | `jsonb` | For dynamic structures. |

## 3. Constraints & Optimization

- **Foreign Keys:** Must follow the pattern `fk_originTable_destinationTable`.
- **Indices:** Mandatory on all foreign key columns and columns frequently used in `WHERE` and `JOIN` clauses.
- **Unique Constraints:** Required for all official catalog codes (`uk_code_mmhh`).
- **Extensions:** The `pgcrypto` extension must be available in the `public` schema.

## 4. Query & Logic Patterns

- **Explicit Joins:** Use explicit `INNER JOIN` or `LEFT JOIN` syntax using defined foreign keys.
- **Null Handling:** Mandatory use of `COALESCE` for numeric fields (e.g., `COALESCE(total_amount, 0)`).
- **Formatting:** Use `UPPER` and `CONCAT` for addresses or legal names in documents.
- **Stored Procedures (PL/pgSQL):** - Use specialized functions for calculations (e.g., `calculo_factura()`).
    - Use `get_comprobante_[type]_electronica()` to generate JSON output.
    - Use `generate_uppercase_uuid()` and `formatear_numero_secuencia()` for IDs and 15-digit sequences.

## 5. Anti-Pattern Detection (Standard Compliance)

- **Do not** allow random UUIDs for primary keys; use the specified integer identity pattern.
- **Do not** leave foreign keys unindexed.
- **Do not** set `spring.jpa.hibernate.ddl-auto` to anything other than `none` in production environments.
- **Do not** use `varchar` without a default empty string if nulls are not permitted.
